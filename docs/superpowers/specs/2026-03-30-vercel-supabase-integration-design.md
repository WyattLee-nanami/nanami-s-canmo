# Vercel + Supabase 集成设计（CanMo）

**Goal:** 将 `client`（Vite React）部署到 Vercel，并把当前 `server`（Express + SQLite）的数据能力迁移到 Supabase（Auth + Postgres + RLS），实现 **每个用户独立数据** 与 **邮箱+密码登录**。

**Architecture:** 前端直连 Supabase（`@supabase/supabase-js`）。Vercel 仅托管前端静态构建产物与路由回退。数据库采用 Supabase Postgres，通过 RLS 保障 `user_id = auth.uid()` 的行级隔离。

**In scope**
- Supabase Auth：邮箱 + 密码注册/登录/登出
- 三张业务表：`meals`、`weight_records`、`goals`（均含 `user_id`）
- RLS 策略：仅允许用户访问自己的行
- 前端 API 适配：将 `client/src/api/client.ts` 从 `fetch('/api/...')` 替换为 Supabase 查询（对上层页面尽量保持接口形状一致）
- Vercel 部署：Root Directory 指向 `client/`，配置 Supabase 环境变量

**Out of scope（本阶段不做）**
- 继续部署当前 `server`（Express）到生产
- 复杂后端业务逻辑（如第三方 API、私密密钥编排）。若后续需要，新增 Vercel Serverless API 作为中间层（方案 2）。
- 多租户、团队协作、管理后台

---

## 1. 当前现状（代码侧）

- `client` 使用 `/api/*` 访问后端（Vite dev 通过代理转发到 `http://localhost:3001`）。
- `server` 为 Express，持久化使用 `better-sqlite3` 本地文件 `canmo.db`，路由提供：
  - `GET/POST/DELETE /api/meals`
  - `GET/POST /api/weight`
  - `GET/PUT /api/goals`
  - `GET /api/stats/daily`、`GET /api/stats/weekly`

本设计将这些数据能力迁移到 Supabase，并在前端复刻 `stats` 聚合逻辑（或通过 SQL view / RPC 进一步优化，见后文可选项）。

---

## 2. 目标架构选型

### 方案 1（采用）：前端直连 Supabase
- **Vercel**：只部署 `client/`（Vite build）
- **Supabase**：Auth + Postgres + RLS
- **客户端**：`@supabase/supabase-js` 负责鉴权与 CRUD

**为什么选它**
- 迁移最少、部署最简单
- 数据隔离由 RLS 保底，风险可控
- 后续需要复杂后端时再引入 Serverless，不阻塞当前上线

### 方案 2（备选）：Vercel Serverless API → Supabase
保留“后端层”，将聚合与校验集中在 Serverless Functions。适合后期复杂逻辑/隐藏密钥/统一审计。

---

## 3. 数据模型（Supabase）

所有表均增加：
- `user_id uuid not null references auth.users(id)`
- RLS：`user_id = auth.uid()`

### 3.1 meals
字段（建议）：
- `id bigint generated by default as identity primary key`
- `user_id uuid not null`
- `food_name text not null`
- `calories integer not null`
- `quantity numeric not null default 1`
- `unit text not null default '份'`
- `meal_type text not null default 'other'`（可做 enum，先用 text 简化）
- `date date not null`
- `created_at timestamptz not null default now()`

索引：
- `(user_id, date)`
- `(user_id, created_at desc)`（可选）

### 3.2 weight_records
字段：
- `id bigint ... pk`
- `user_id uuid not null`
- `weight numeric not null`
- `date date not null`
- `created_at timestamptz not null default now()`

约束/索引：
- `unique(user_id, date)`
- `(user_id, date)`

### 3.3 goals
字段：
- `id bigint ... pk`
- `user_id uuid not null`
- `daily_calories integer not null default 1800`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

约束：
- `unique(user_id)`（每用户 1 行）

初始化策略：
- 用户首次进入时，如果没有 goals 行，前端 `upsert` 默认 `daily_calories=1800`。

---

## 4. 安全与权限（RLS）

对 `meals` / `weight_records` / `goals`：
- Enable RLS
- Policies（概念）：
  - **SELECT**：`user_id = auth.uid()`
  - **INSERT**：`user_id = auth.uid()`
  - **UPDATE**：`user_id = auth.uid()`
  - **DELETE**：`user_id = auth.uid()`

注意：
- 前端写入时必须显式写 `user_id`（或用数据库默认值 + trigger，先不做 trigger，保持简单）。
- 后续如要避免客户端写 `user_id`，可引入 `auth.uid()` 默认列/触发器或 RPC。

## 5. 前端改造设计（client）

### 5.1 Supabase 客户端
新增模块：
- `client/src/lib/supabase.ts`
  - 读取 `import.meta.env.VITE_SUPABASE_URL`、`VITE_SUPABASE_ANON_KEY`
  - 导出 `supabase` 实例

环境变量：
- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`

### 5.2 登录/注册/路由保护
新增页面：
- `client/src/pages/AuthPage.tsx`（登录 + 注册）

路由保护（按现有路由结构选择最小改动方式）：
- 若当前是手写路由/组件切换：在顶层 `App` 或 `Layout` 判断 session，未登录就渲染 `AuthPage`。
- Session 获取：
  - `supabase.auth.getSession()`
  - `supabase.auth.onAuthStateChange(...)` 订阅并更新 UI

### 5.3 API 层迁移：保持上层调用不变
修改 `client/src/api/client.ts`：
- 用 Supabase 替代 `fetch('/api/...')`
- `mealsApi` / `weightApi` / `goalsApi` / `statsApi` 保持同名导出

`statsApi` 计算方式：
- `getDaily(date)`：查询当天 meals + goals，前端计算 `totalCalories/remaining`
- `getWeekly()`：循环 7 天分别聚合（先简单实现），后续可优化为单条 SQL group by 或 view/RPC

错误处理：
- 将 Supabase error 转为 `throw new Error(message)`，保持页面处理方式一致

## 6. Vercel 部署设计（client）

在 Vercel 导入 GitHub 仓库后：
- **Root Directory**：`client`
- **Build Command**：`npm run build`
- **Output**：Vite 默认（Vercel 能识别）
- **Env Vars**：
  - `VITE_SUPABASE_URL`
  - `VITE_SUPABASE_ANON_KEY`

若使用前端路由（History API）：
- 需要 Vercel 的 SPA 回退配置（`vercel.json` rewrite 到 `/index.html`），确保刷新子路由不 404。

---

## 7. 迁移与回滚策略

迁移：
- 先上线 Auth + 空数据流程
- 再迁移 CRUD
- 最后替换 stats 聚合

回滚：
- 由于不改数据库（Supabase 独立），回滚主要是前端版本回滚到上一部署产物即可。

---

## 8. 验收标准（Definition of Done）

- Vercel 生产环境可访问，未登录会提示登录/注册
- 注册后可登录并保持会话刷新不丢失
- 用户 A 与 用户 B 的 meals/weight/goals 互不可见（RLS 生效）
- 首页今日摘要/周统计可正常加载（与旧逻辑一致或合理等价）
- `client` 不再依赖本地 `/api` 代理即可工作
